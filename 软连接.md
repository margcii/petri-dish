🧠 全局开发日志软链接工作流
1. 架构逻辑：物理分散，逻辑聚合
通过 Windows 目录联接（Junction Point），将各个独立项目的文档文件夹，统一“映射”到 Obsidian 的指定枢纽中。
执行层（物理层）：项目A/docs、项目B/docs（Agent的施工区，各自独立，Git 隔离）
透视层（逻辑层）：Obsidian库/record/devlog（人类的监控室，全局聚合）
唯一事实来源 (SSOT)：底层共用同一份 .md 文件，任何一端修改，双向实时同步。
2. 核心操作：建立软链接
在新增任何项目时，必须执行此操作将日志接入系统。
工具：管理员模式打开 CMD / PowerShell
公式：mklink /J "Obsidian目标路径" "项目源路径"
实战示例（将项目 A 和 B 的 docs 接入 record\devlog）：
code
Powershell
# 1. 接入项目A (注意重命名目标文件夹，防止同名冲突)
mklink /J "D:\ObsidianVault\record\devlog\ProjectA_docs" "C:\Projects\ProjectA\docs"

# 2. 接入项目B
mklink /J "D:\ObsidianVault\record\devlog\ProjectB_docs" "C:\Projects\ProjectB\docs"
3. Obsidian 白板工作流 (Canvas Workflow)
建立映射后，文件管理器操作降级，一切以 Canvas 视觉交互为主。
构建中控台 (Master Dashboard)：
在 Obsidian 根目录创建 Master_Board.canvas。
组件装配：
将 record\devlog\ProjectA_docs\sprint.md（进度表）和 architecture.md（架构图）直接拖入白板。
拖入后，它们会以实时卡片的形式展现。
人类操作规范：
查看：通过缩放白板俯瞰所有项目的当前节点。
编辑：双击 Canvas 里的卡片直接修改。修改会瞬间穿透回 C 盘的项目源码中。
连线：仅在白板层面进行跨卡片的逻辑连线，不污染底层 Markdown。
4. 协作规约 (Human-Agent Contract)
角色	交互界面	核心职责	权限与限制
人类 (你)	Obsidian Canvas	制定愿景、划定 Sprint、跨项目复盘	只在白板中编辑，不直接打开具体项目的 .md。
Agent (Cline)	VS Code 工作区	阅读 docs/*.md 获取指令，完成任务后更新文件	对软链接和白板零感知。必须在 .clinerules 强制其改写状态。
版本控制 (Git)	项目根目录	将代码与项目本地的 docs 一同 Commit	保持干净，Obsidian 的配置和白板结构不会被推送到项目仓库中。
5. 避坑指南
删除需谨慎：如果要在 Obsidian 中移除某个项目的映射，只删除 Obsidian 里的那个快捷方式文件夹，千万不要在 Obsidian 里删除文件夹内的 .md，否则原项目的物理文件也会被删！
相对路径陷阱：项目 .md 中的图片引用应放在该项目的 docs/assets 下。不要引用 Obsidian 库里其他地方的图片，否则 Agent 在 VS Code 里会读不到（因为超越了项目作用域）。
